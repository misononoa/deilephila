# MVP スコープ・マイルストーン・未解決論点

## 1. MVP の目標

「2人以上のユーザーが、異なるIPネットワークに接続されたそれぞれのマシンから、サーバーなしで、テキスト投稿を作成し、フォロー相手のタイムラインを見られる」ことを最小の成功条件とする。

## 2. MVP に含む / 含まない

### 含む(In Scope)
- 鍵ペア生成によるアカウント作成、秘密鍵のローカル暗号保管
- プロフィール(表示名・bio)の設定と公開
- テキスト投稿の作成(署名 + IPFSチェーンへの追記)
- 公開鍵によるフォロー、フォロー相手のタイムライン受信・表示
- gossipsub によるリアルタイム新着 + DHT/直接問い合わせによる取りこぼし回収
- LAN内(mDNS)およびブートストラップ経由のピア発見
- ローカルSQLiteによるキャッシュ・タイムライン表示

### 含まない(Out of Scope, 将来)
- 画像・メディア添付
- 返信・スレッド・いいね/リアクション
- フォロワー数の正確な集計
- 高度なモデレーション(共有ブロックリスト、Web of Trust)
- 複数デバイス間の鍵・状態同期
- モバイル対応

## 4. 未解決論点

### R1. head 発見 — 確定(D6 / [networking.md](networking.md) §4)

チェーンの最新イベントを指す可変ポインタは IPNS-headレコードに一本化した(IPNS名 = アカウント公開鍵)。当初は「自前の mutable record を設計するか、IPNS を採用するか」が要決定事項だったが、IPNS の採用で決着している。

- 搬送経路: 署名付き IPNS-headレコードを gossipsub と DHT の両方で配信する。gossipsub はオンラインのフォロワーへの即時通知を、DHT はオフラインだったピアが後から取得するための永続配信を担う。どちらの経路でも取得できない場合や、取得できたレコードの鮮度が疑わしい場合は、フォローグラフ内のピアに `GetLatestHead` で問い合わせて、より良いレコードを探索する。
- ワイヤ形式: レコードはイベントチェーンと同じ canonical DAG-CBOR でシリアライズする。IPNS 仕様の protobuf 形式(`IpnsEntry`)は、公開 IPFS との相互運用を必要としない([networking.md](networking.md) §3.1)一方で V1/V2 二重署名や protobuf 依存の追加など実装コストが大きいため採用しない。
- 解決規則: 複数の候補レコードが集まったとき、「署名検証に成功し、かつ (sequence, validity) が辞書式で最大のもの」を採用する(argmax統一規則)。経路ごとに優先順位を付ける階層的フォールバックではなくこの単一規則に統一することで、古い(stale な)レコードに解決結果が引きずられることを防ぐ。validity を副キーにするのは、republish が `sequence` を変えず validity のみ更新するため(当初は `sequence` 単独比較だったが、それでは republish が全受信側で stale と判定され、レコードの生存期間を延長できないことが判明し変更した)。
- 安全性: チェーンは追記専用かつ全イベントが署名付きなので、第三者が本人になりすまして偽の新イベントを作ること(forward-forge)はできない。攻撃者にできるのは、古いレコードだけを提示して最新の更新を隠すこと(出し惜しみ)までであり、それも argmax統一規則により、より新しいレコードが1つでも届いた時点で無効化される。
- 原理的な限界: アカウントが新規で誰にも複製されておらず、オンラインのフォロワーが1人もおらず、かつ IPNS-headレコードが EOL を過ぎて DHT から失効している、という3条件が重なると、そのアカウントの head は発見できない。これはサーバーを持たない純粋な P2P 構成の原理的な床であり、受容する。

### R2. データ可用性・複製 — 確定(D8 / [networking.md](networking.md) §3.2)

R1 と密結合の論点である。head ポインタ(IPNS-headレコード)もイベントのブロック本体も、同じフォロワー集合が保持・提供することで「可用性ネットワーク」の役割を担う。何をどれだけ複製するかはクライアントごとの任意設定とする。この方針が安全なのは、整合性(データが本物であること)は署名と CID によって常に保証され、複製ポリシーに依存しないためである。一方、可用性(データが取得できること)はフォロワーたちの保持状況から創発するものであり、保証はしない。この非対称を受容するのが設計の骨子である。

- 設定軸: 複製対象とするアカウントの範囲 / アカウントごとの保持量 / ストレージ全体の上限 / 明示的なピン留め、の4軸を設ける。MVP ではアカウント単位の指定のみを実装する。
- デフォルトポリシー: フォロー相手のチェーンを、全体上限の範囲内で LRU(最近使われていないものから追い出す)方式で複製する。ただし各フォロー相手の最新 IPNS-headレコードだけは、ブロック本体を追い出した後も常に保持し続ける。ポインタとブロックの保持を分離することで、D7 および R1 で述べた発見可能性の床を確保するためである。自分のチェーンは常にピン留めし、追い出しの対象にしない。保持しているデータはブロック交換プロトコルで他のピアに提供する。
- チェーン途中の取得できない gap は設計上受容し、同期エンジン側も同期窓・チャンク挿入・再開カーソルによる部分同期でこれに対応する([networking.md](networking.md) §4.4、[data-model.md](data-model.md) §6)。全件取得できるまで何も取り込まない、という動作はしない。各投稿は単独で署名検証できるため、gap があっても整合性そのものは損なわれない。
- 残課題: チェーンの一部しか保持しないピアが増えると、equivocation(同一 `seq` で異なるイベントを別々のピアに配る二枚舌)の検出力が下がる。また、常時稼働のピンニングノードを任意で併用するハイブリッド構成を許容するかは将来の検討事項とする。

### R3. 編集・削除とプライバシー

データ構造としては確定済みである(D4)。編集・削除は `Edit` / `Delete` イベントとして同一チェーンに追記し、過去のイベントは上書きしない([data-model.md](data-model.md) §2.3・§4)。fold の際に、編集は最後の `Edit` が有効(last-write-wins)、削除された投稿は非表示として扱う。

- 未解決なのはプライバシー方針である。チェーンは追記専用であり、IPFS のブロックは不変なので、編集前の本文や削除済み投稿の実データは、それを保持しているピアのディスク上に残り続けうる。これを踏まえて、削除が論理削除にすぎないことを UI 上でどう明示するか、また `Delete` イベントをまだ受け取っていないピアが削除済み投稿を再配信してしまうことをどこまで抑止するか(および、その抑止にどれだけの実装コストを掛けるか)を決める必要がある。
- より根本的には、検閲耐性(誰も他人の発言を消せないこと)と忘れられる権利(自分の発言を取り消せること)は原理的にトレードオフの関係にある。どちらをどこまで優先するかをプロダクト方針として明文化する。

### R4. スパム・モデレーション

基本方針は次の組み合わせとする([networking.md](networking.md) §6)。まず、タイムラインには自分がフォローした相手の投稿しか流れないため、購読ベースの構造そのものが自然なスパムフィルタとして働く。そのうえで、署名検証による偽イベントの排除、gossipsub の peer scoring による不正ピアのメッシュからの除外、ユーザー自身によるローカルブロック(特定アカウントの非表示・受信拒否)を組み合わせる。

- 要決定: アカウント(鍵ペア)の大量生成による濫用を抑止するために、鍵生成に PoW 的な計算コストを課すか否か。MVP では課さない方針とし、必要性が見えた段階で再検討する。

### R5. 鍵管理・リカバリ

アイデンティティが Ed25519 鍵ペアそのものであるため、秘密鍵を失うことはアカウントの復旧手段が一切ないことを意味する。これはサーバーに頼らない自己主権型アイデンティティの代償として受容するが、緩和策の設計が論点として残っている。

- 要決定: ニーモニックフレーズ(BIP39 風の単語列)による秘密鍵のバックアップ手段を提供するか。また、複数デバイスで同一アカウントを使うための鍵・状態の同期方式をどうするか(いずれも MVP の範囲外)。
- 鍵が漏洩した場合のローテーション手段も将来課題である。想定している方向性は、新しい鍵ペアを生成し、旧鍵による署名付きの移行宣言をチェーンに載せることで、フォロワーに鍵の切り替えを伝える方式である。
